# Feature Specification: Reports, Summaries, and History

**Feature Branch**: `009-reports-summaries-history`

**Created**: 2026-07-08

**Status**: Draft

**Input**: User description: "Phase 9 — Reports, Summaries, and History. Finalize the MVP reporting surface, add lightweight history/activity tracking, and add optional AI spending summaries. Builds on the existing Phase 4/5 dashboard and reports page and must extend, not rebuild, that surface."

## Overview

This feature finalizes the MVP reporting surface, adds a lightweight workspace
activity history, and adds an optional AI-generated spending summary. It builds
on the dashboard and reports surfaces already delivered in earlier phases (which
present the current-period summary: total income, total expenses, remaining
balance, category breakdown, recent records, and pending AI review count) and
extends them so users can choose a reporting period, see spending trends and top
merchants, review team activity, read a plain-language spending summary, and — if
they enabled AI — read an optional AI spending summary.

All reporting figures are derived from **confirmed records only** and must
reconcile with the dashboard. The activity history is practical traceability, not
an audit-grade ledger. AI features remain fully optional; the entire feature
except the optional AI summary works with no AI key configured.

## Clarifications

### Session 2026-07-08

- Q: What is the maximum span allowed for a custom reporting date range? → A: 366 days (one year); ranges longer than 366 days are rejected as invalid.
- Q: When does the spending trend use daily vs monthly granularity? → A: Daily granularity when the selected period spans 31 days or fewer; monthly granularity when it spans more than 31 days.
- Q: How is the optional AI spending summary triggered? → A: On-demand only — generated by an explicit user action, never automatically on page load (avoids consuming BYOK tokens without intent).
- Q: What does the team activity summary contain in the MVP? → A: Per-member counts of confirmed records (income + expenses) created within the selected period, derived from confirmed records and scoped to the workspace and period.
- Q: What are the history retention and view limits? → A: Retain all forward events with no automatic purge in the MVP; the activity history view is paginated newest-first with a default page size of 50.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - View a full spending report for a chosen period (Priority: P1)

A workspace member opens the reports page, selects a reporting period (for
example the current month, last month, or a custom date range), and sees a
consolidated report: income vs. expenses, remaining balance for the period,
category breakdown, a daily or monthly spending trend, top merchants, and recent
records (recent income and expenses). Every figure is based only on confirmed
records and matches what the dashboard shows for the same period.

**Why this priority**: Reports are the headline deliverable of this phase and the
primary value to users — being able to answer "where did my money go this period"
beyond the current-month dashboard snapshot. It is independently shippable and
delivers value on its own.

**Independent Test**: Seed a workspace with confirmed income and expenses across
two months plus some deleted and some draft/pending AI-extracted records. Open
reports, select each period, and verify the totals, category breakdown, trend,
and top merchants reflect only confirmed records for that period and equal the
dashboard totals for the same period.

**Acceptance Scenarios**:

1. **Given** a workspace with confirmed income and expenses in the current month,
   **When** the user opens reports with the default (current) period, **Then**
   income vs. expenses, remaining balance, category breakdown, spending trend,
   top merchants, and recent records are shown and the income/expense/remaining
   totals equal the dashboard totals for the current period.
2. **Given** the user is viewing reports, **When** they change the reporting
   period to a prior month or a custom date range, **Then** all report figures
   recompute for the selected period.
3. **Given** the workspace has draft, pending, failed, cancelled, or deleted
   records, **When** any report is computed, **Then** those records are excluded
   from every figure.
4. **Given** expenses that have a merchant name and expenses that do not, **When**
   the top-merchants report is computed, **Then** only expenses with a merchant
   name are aggregated into merchant totals and expenses without a merchant name
   are not misattributed.
5. **Given** a workspace with no confirmed records in the selected period, **When**
   the user views reports, **Then** each report shows a clear empty state rather
   than an error or a misleading zero-as-data chart.

### User Story 2 - Read a plain-language spending summary (Priority: P2)

On the reports page, the user sees a short, plain-language summary of their
spending for the selected period — for example total income, total expenses,
remaining balance, the top spending category, and whether spending went up or
down versus the previous comparable period. The summary is understandable to
users without an accounting background and is available in both Arabic and
English. This summary is computed from the confirmed data and requires no AI key.

**Why this priority**: The plain-language summary turns raw report figures into an
at-a-glance narrative for non-accounting users, reinforcing the Saudi-first,
simple-language product goal. It depends on the period-scoped figures from Story 1
but adds distinct value.

**Independent Test**: Seed known confirmed data for a period and the prior
comparable period, open reports, and verify the summary states the correct
totals, correct top category, and correct up/down trend direction, rendered
correctly in both English and Arabic (RTL).

**Acceptance Scenarios**:

1. **Given** confirmed income and expenses in the selected period, **When** the
   summary renders, **Then** it states total income, total expenses, remaining
   balance, and the top spending category for that period in plain language.
2. **Given** the previous comparable period had different total spending, **When**
   the summary renders, **Then** it states whether spending increased, decreased,
   or stayed roughly the same versus that prior period.
3. **Given** the UI language is Arabic, **When** the summary renders, **Then** it
   is shown in Arabic with correct RTL layout and SAR formatting.
4. **Given** no confirmed records exist in the period, **When** the summary
   renders, **Then** it shows a neutral empty-state message and no misleading
   figures.

### User Story 3 - View workspace activity history (Priority: P2)

An Owner or Admin opens the activity history for the workspace and sees a
reverse-chronological list of recent workspace events — records created, updated,
and deleted; files uploaded and deleted; AI extraction started, completed, and
failed; AI drafts confirmed; workspace members added and removed; roles changed;
and settings changed. Each entry shows what happened, who did it, and when. The
history is practical traceability, not a full audit ledger, and only records
events that occur from this phase forward.

**Why this priority**: History gives teams and individuals confidence that they
can see what changed and by whom, which is important for shared/team workspaces
and for debugging surprises in totals. It is the main net-new data surface of this
phase and is independently testable.

**Independent Test**: As an Owner, perform a series of actions (add an expense,
edit it, delete it, upload a file, confirm an AI draft, change a member role,
change a setting) and verify each produces exactly one corresponding history entry
with the correct actor, event type, and timestamp, newest first. Verify a Member
and a Viewer cannot open the history view.

**Acceptance Scenarios**:

1. **Given** a workspace, **When** a tracked action occurs (e.g. an expense is
   created), **Then** a single history entry is recorded synchronously with the
   event type, the acting user, an entity reference, and a timestamp.
2. **Given** history entries exist, **When** an Owner or Admin opens the activity
   history, **Then** entries are listed newest-first and each shows what happened,
   who did it, and when.
3. **Given** the workspace existed before this phase with prior records, **When**
   history is viewed, **Then** only events that occurred from this phase forward
   appear (no backfilled historical events), and this is not treated as an error.
4. **Given** a user with the Member or Viewer role, **When** they attempt to open
   or request the activity history, **Then** access is denied by the backend.
5. **Given** a user who is not a member of the workspace, **When** they attempt to
   read that workspace's history, **Then** access is denied.

### User Story 4 - See team activity and pending review summaries (Priority: P3)

A workspace member viewing reports sees a team activity summary (a compact
overview of recent workspace activity, such as counts of records added by members
in the period) and a pending invoice review summary (how many AI extraction
results are still awaiting review). These help teams understand collaboration and
outstanding work without opening the full history.

**Why this priority**: These summaries round out the constitution-mandated report
list and are useful for team workspaces, but they are secondary to the core
period report and depend on the history/extraction data being available.

**Independent Test**: Seed a workspace with multiple members adding confirmed
records in a period and some AI extractions left in pending/review state; open
reports and verify the team activity summary reflects the members' contributions
for the period and the pending review summary shows the correct count of items
awaiting review.

**Acceptance Scenarios**:

1. **Given** multiple members added confirmed records during the selected period,
   **When** the team activity summary renders, **Then** it reflects that activity
   scoped to the period and the workspace.
2. **Given** some AI extraction results are pending review, **When** the pending
   review summary renders, **Then** it shows the count of items still awaiting
   review, consistent with the dashboard's pending review count.
3. **Given** a personal (single-member) workspace, **When** the team activity
   summary renders, **Then** it degrades gracefully (e.g. shows the single user's
   activity or a neutral state) rather than erroring.

---

### User Story 5 - Read an optional AI spending summary (Priority: P3)

A workspace that has configured a BYOK AI key can request an AI-generated spending
summary for the selected period. The AI summary uses only that workspace's
confirmed data, is written in plain language, avoids financial advice, and is
available in Arabic and English. If no AI key is configured, the AI summary simply
does not appear and everything else on the page still works.

**Why this priority**: This is a pure convenience layer on top of the confirmed
data and is strictly optional per the manual-first / AI-optional principle. It is
the lowest priority because the product must be fully useful without it.

**Independent Test**: With no AI key configured, open reports and confirm no AI
summary is offered and the page works fully. Then configure a valid BYOK key,
request an AI summary, and confirm it returns a plain-language summary derived only
from the workspace's confirmed data; simulate a provider failure and confirm a
safe error with the rest of the page unaffected.

**Acceptance Scenarios**:

1. **Given** the workspace has no AI key configured, **When** the reports page
   loads, **Then** no AI summary is shown and no AI summary action is offered, and
   all non-AI reports and summaries work normally.
2. **Given** the workspace has a valid AI key configured, **When** an Owner,
   Admin, or Member requests an AI spending summary for the period, **Then** a
   plain-language summary based only on that workspace's confirmed data is returned
   in the current UI language.
2a. **Given** the workspace has a valid AI key configured, **When** a Viewer views
   the reports page, **Then** the AI summary action is not offered to them and all
   non-AI reports and summaries remain fully available.
3. **Given** the AI provider fails or the key is invalid, **When** the user
   requests an AI summary, **Then** a safe, non-technical error is shown and the
   rest of the reports page continues to function.
4. **Given** an AI summary is generated, **When** it is displayed, **Then** it does
   not contain financial advice and does not reference data outside the workspace.

### Edge Cases

- **Empty period**: A selected period with no confirmed records shows empty states
  for every report and summary, never an error or misleading chart.
- **Period with only income or only expenses**: Reports render correctly when one
  side is zero (e.g. remaining balance equals income; expense breakdown empty).
- **Negative remaining balance**: When confirmed expenses exceed confirmed income
  for the period, remaining balance is shown correctly as negative.
- **Custom date range validation**: An invalid range (end before start, or a span
  exceeding 366 days) is rejected rather than producing a broken report.
- **Expenses without a merchant name**: Excluded from top-merchants aggregation
  and not grouped under a blank/"unknown" bucket that distorts totals.
- **Expenses without a category**: Represented in the category breakdown under a
  clear "uncategorized" grouping rather than being dropped from expense totals.
- **Daily vs monthly trend granularity**: A short period uses daily granularity; a
  long period uses monthly granularity, so trend charts stay readable.
- **Deleted records after report view**: Deleting a record and reopening the report
  reflects the change (reports are computed live, not cached stale).
- **Timezone/date boundaries**: Records are attributed to a period by their
  business date consistently, so a record does not appear in two periods or none.
- **History for high-frequency actions**: Bulk or rapid actions each still produce
  correct individual history entries without loss or duplication.
- **AI summary requested for an empty period**: Returns a neutral message rather
  than fabricating figures.
- **Viewer role**: Can view reports and summaries but cannot view history and
  cannot perform any write action anywhere in the feature.

## Requirements *(mandatory)*

### Functional Requirements

#### Reports — scope, period, and integrity

- **FR-001**: System MUST provide a reports view that presents, for a selected
  reporting period: income vs. expenses, remaining balance, category breakdown,
  a daily-or-monthly spending trend, top merchants, and recent records (recent
  income and expenses, matching the dashboard's recent-activity list).
- **FR-002**: System MUST allow the user to select the reporting period, including
  at minimum the current month, a previous month, and a custom start/end date
  range, defaulting to the current period.
- **FR-003**: System MUST compute every report figure from confirmed records only
  (confirmed income and confirmed expenses), excluding records in a deleted state.
- **FR-004**: System MUST exclude all AI extraction records that are in draft,
  pending, processing, failed, or cancelled states from every report figure; AI
  data affects reports only after it has become a confirmed expense.
- **FR-005**: Report income, expense, and remaining-balance totals MUST reconcile
  with the dashboard totals for the same workspace and equivalent period (same
  confirmed-only filter and same money handling), so figures cannot drift between
  the two surfaces.
- **FR-006**: System MUST compute and store all money amounts using integer minor
  units (or fixed-precision decimals) and MUST NOT use floating-point arithmetic
  for any financial figure in reports or summaries.
- **FR-007**: System MUST compute the category breakdown for the period, including
  a clearly labelled grouping for expenses that have no category, without dropping
  those expenses from the expense total.
- **FR-008**: System MUST compute top merchants for the period from expenses that
  have a merchant name, and MUST exclude expenses without a merchant name from
  merchant aggregation.
- **FR-009**: System MUST compute a spending trend for the period, using daily
  granularity when the selected period spans 31 days or fewer and monthly
  granularity when it spans more than 31 days, so the trend remains readable.
- **FR-010**: System MUST compute a remaining-balance figure/trend for the period
  consistent with the core formula: remaining balance = confirmed income −
  confirmed expenses.
- **FR-011**: System MUST show a clear empty state for each report when the
  selected period contains no relevant confirmed records, rather than an error or
  a misleading chart.
- **FR-012**: System MUST validate the selected reporting period and reject
  invalid ranges: an end date before the start date, and any custom range spanning
  more than 366 days, MUST be rejected rather than producing a broken report.

#### Plain-language spending summary

- **FR-013**: System MUST present a plain-language spending summary for the
  selected period stating total income, total expenses, remaining balance, and the
  top spending category, understandable without an accounting background.
- **FR-014**: The plain-language summary MUST indicate whether spending increased,
  decreased, or was roughly unchanged versus the previous comparable period.
- **FR-015**: The plain-language summary MUST be available in both Arabic and
  English with correct RTL layout and SAR formatting, and MUST require no AI key.

#### Team activity and pending review summaries

- **FR-016**: System MUST present a team activity summary for the selected period
  consisting of per-member counts of confirmed records (income and expenses)
  created within that period, derived from confirmed records and scoped to the
  workspace and period.
- **FR-017**: System MUST present a pending invoice review summary showing the
  count of AI extraction results still awaiting review, consistent with the
  dashboard's pending review count.
- **FR-018**: The team activity summary MUST degrade gracefully for personal
  (single-member) workspaces rather than erroring.

#### History / activity tracking

- **FR-019**: System MUST record a workspace activity history entry when any of the
  following events occurs: income created/updated/deleted; expense
  created/updated/deleted; category created/updated/archived; file uploaded/
  deleted; AI extraction started/completed/failed; AI draft confirmed; workspace
  member added/removed; role changed; setting changed.
- **FR-020**: Each history entry MUST capture the event type, the acting user, a
  reference to the affected entity, the workspace it belongs to, and a timestamp.
- **FR-021**: History entries MUST be written synchronously as part of the action
  that triggers them, so a successful action is reliably reflected in history.
- **FR-022**: History MUST be forward-only: only events occurring from this feature
  onward are recorded; pre-existing records MUST NOT be backfilled, and their
  absence MUST NOT be treated as an error.
- **FR-023**: History MUST be a lightweight traceability log and MUST NOT implement
  audit-grade immutability, field-level diffs/versioning, or ledger semantics.
- **FR-024**: System MUST present the activity history newest-first, showing for
  each entry what happened, who did it, and when; the view MUST be paginated with a
  default page size of 50 entries.
- **FR-024a**: System MUST retain all recorded history events for the workspace
  with no automatic purge or retention window in the MVP.
- **FR-025**: Every history entry MUST belong to exactly one workspace, and history
  MUST be readable only by authorized members of that workspace.

#### Optional AI spending summary

- **FR-026**: System MUST offer an AI-generated spending summary only when the
  workspace has a valid BYOK AI key configured; when no key is configured, the AI
  summary MUST be absent and all other functionality MUST remain fully usable.
- **FR-026a**: The AI spending summary MUST be generated on demand only, triggered
  by an explicit user action, and MUST NOT be generated automatically on page load.
- **FR-027**: The AI spending summary MUST be generated using only the requesting
  workspace's authorized data and MUST be based on confirmed records.
- **FR-028**: The AI spending summary MUST use plain language, MUST NOT provide
  financial advice, and MUST support Arabic and English.
- **FR-029**: When AI summary generation fails (invalid key or provider error), the
  system MUST show a safe, non-technical error and MUST NOT break the rest of the
  reports page.

#### Permissions and tenant isolation

- **FR-030**: System MUST enforce access to all reports, summaries, and history on
  the backend and at the database policy level by workspace membership, not by
  frontend checks alone.
- **FR-031**: Reports and the non-AI summaries (the plain-language spending
  summary, the team activity summary, and the pending review summary) MUST be
  viewable by all workspace roles, including Viewer. The on-demand AI spending
  summary is an outbound action that consumes the workspace's BYOK key, so
  initiating it MUST be restricted to Owner, Admin, and Member roles (the same set
  authorized to use the workspace AI key); Viewer MUST NOT be able to initiate it.
- **FR-032**: The activity history view MUST be restricted to Owner and Admin
  roles; Member and Viewer roles MUST NOT be able to read it.
- **FR-033**: No role in this feature grants any write capability through the
  reports, summaries, or history surfaces; Viewer MUST remain strictly read-only,
  and users MUST NOT be able to access another workspace's reports, summaries, or
  history.

#### Out of scope (explicitly deferred)

- **FR-034**: The following MUST NOT be implemented in this phase: report export to
  PDF or CSV; scheduled or emailed reports; stored/persisted report snapshots;
  audit-grade or compliance history; cross-workspace/aggregate reporting; and
  multi-currency reporting (reporting is SAR-first only).

### Key Entities *(include if feature involves data)*

- **Report (computed, not stored)**: A period-scoped, on-demand aggregation over a
  workspace's confirmed income and expenses. Attributes conceptually include the
  selected period, income total, expense total, remaining balance, category
  breakdown, spending trend series, top merchants, and recent records (income and
  expenses). Not persisted (no snapshots in MVP).
- **Spending Summary (computed, not stored)**: A plain-language derivation from the
  Report figures for the period, including totals, top category, and trend
  direction versus the prior comparable period. Localized (Arabic/English).
- **Team Activity Summary (computed, not stored)**: A period-scoped, workspace-
  scoped overview of member activity, expressed as per-member counts of confirmed
  income and expense records created within the period.
- **Pending Review Summary (computed, not stored)**: A count of AI extraction
  results awaiting review, consistent with the dashboard.
- **Activity History Entry (stored)**: A workspace-scoped, forward-only record of a
  tracked event. Attributes: workspace reference, event type, acting user, affected
  entity reference, and timestamp. Retained with no automatic purge; read-restricted
  to Owner/Admin; presented newest-first with pagination (default page size 50).
- **AI Spending Summary (computed, not stored)**: An optional, BYOK-gated, plain-
  language summary generated from the workspace's confirmed data for the period.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For any selected period, the report's total income, total expenses,
  and remaining balance exactly match the dashboard's totals for the same workspace
  and equivalent period (zero discrepancy) in 100% of test cases, including cases
  with deleted, draft, and pending AI records present.
- **SC-002**: 100% of draft, pending, processing, failed, cancelled, and deleted
  records are excluded from every report and summary figure.
- **SC-003**: A user can change the reporting period (current month, a prior month,
  or a custom range) and see all report figures recompute for that period in a
  single interaction, with no page reload of unrelated context.
- **SC-004**: 100% of tracked actions produce exactly one corresponding history
  entry (no missing and no duplicate entries) with the correct actor, event type,
  and timestamp.
- **SC-005**: Members and Viewers are denied access to the activity history in 100%
  of attempts, and no user can access another workspace's reports, summaries, or
  history in 100% of cross-workspace attempts.
- **SC-006**: With no AI key configured, 100% of non-AI reports and summaries render
  and function normally, and no AI summary is presented.
- **SC-007**: The plain-language spending summary renders correctly in both English
  (LTR) and Arabic (RTL) with SAR formatting for all tested periods.
- **SC-008**: When the AI provider fails or the key is invalid, the AI summary
  request returns a safe, non-technical error and the rest of the reports page
  remains fully functional in 100% of such cases.
- **SC-009**: Every report and summary correctly renders an empty state (no error,
  no misleading chart) for a period containing no confirmed records.

## Assumptions

- The dashboard and reports surfaces from earlier phases exist and expose the
  current-period confirmed-only totals; this phase extends those surfaces rather
  than rebuilding them, and reuses the same confirmed-only aggregation semantics.
- Confirmed records are those with a confirmed status; both income and expense
  records carry a status distinguishing confirmed from deleted, and expenses carry
  an optional merchant name and optional category, per the existing data model.
- The reporting period is attributed using each record's business/occurrence date.
- "Top merchants when available" means merchant totals are derived from the
  merchant name already captured on expenses; no new merchant catalog is
  introduced.
- History is stored in a new workspace-scoped table introduced in this phase and is
  protected by row-level security; the plan/data-model will describe a properly
  tracked database migration for it (no application code is written in this
  planning phase).
- The team activity summary in the MVP is a simple period-scoped overview derived
  from confirmed records and/or history, not a per-user analytics product.
- AI spending summaries reuse the existing BYOK AI configuration and provider
  integration from earlier phases; this phase adds only the summary use case, gated
  on a configured key.
- Reports are computed live on request; no report snapshots are persisted in the
  MVP.
- SAR is the only reporting currency in the MVP; multi-currency reporting is out of
  scope.
